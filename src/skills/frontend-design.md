# {{SKILL}}: frontend-design

> {{DESC_FRONTEND_DESIGN}}

**Trigger:** building a new UI or reshaping an existing one, and it should carry a
deliberate point of view rather than read as a templated default.

## Procedure
1. Ground it first: name the concrete subject, the audience, and the single job
   this page does. Draw distinctive choices from the subject's own world — its
   materials, instruments, artifacts, and vernacular — and build with real
   content, not lorem.
2. First pass — write a compact design plan before any code:
   - **Color:** 4–6 named hex values, each with a description.
   - **Type:** a characterful display face used sparingly, a complementary body
     face, and a utility face for captions/data if needed — paired deliberately,
     not your default reach.
   - **Layout:** one-sentence prose plus an ASCII wireframe per section.
   - **Signature:** the single element that embodies this brief.
3. Open with the hero as thesis: the most characteristic thing in the subject's
   world (headline, image, animation, live demo), not a templated big-number row.
4. Make structure earn its place — numbered markers (01/02/03) only for genuine
   sequences; cut any device that decorates without encoding information. Deploy
   motion only where it serves the subject; scattered animation reads as
   AI-generated.
5. Avoid the three tells unless the brief truly calls for them: (a) cream
   `#F4F1EA` + high-contrast serif + terracotta accent; (b) near-black + acid-green
   or vermilion accent; (c) broadsheet hairline rules, zero radius, dense columns.
6. Second pass — critique against the brief ({{LAW}} III): screenshot the result
   and read it. If any part reads like the generic default you'd produce for any
   similar page, revise it. Spend boldness in one place and keep the rest quiet;
   remove one accessory. Hold the quality floor without announcing it — responsive,
   visible keyboard focus, reduced-motion support.
7. Treat UI copy as design material, not decoration: end-user's viewpoint, active
   voice, one name per action through the whole flow ("Publish" → "Published").
   Errors explain the fix; empty states invite action. Run the
   [prose {{SKILL}}](../{{DIR_SKILLS}}/prose.md) over the words.

## Done when
- The design makes one justified aesthetic risk specific to this brief, carries no
  templated tells, holds the accessibility floor, and a screenshot critique finds
  nothing that reads as a generic default.

## Self-improvement

Close each run with one beat of reflection on the {{SKILL}} itself:
- A step misled, a needed step was missing, or the trigger fired wrongly — that
  is a flaw in this file. Propose the exact edit (trigger, procedure, or
  done-when) and apply it with the user's assent ({{LAW}} II).
- A lesson that is *not* a flaw in this file goes to {{MEMORY}} only if it
  clears {{LAW}} VI's bar: it would change how a future session behaves, and a
  fresh read of the repo would not re-derive it. Update an existing memory over
  adding one; when in doubt, leave it out.
- No friction, nothing learned — move on; this loop earns no ceremony. Most
  runs end here.
