# {{SKILL}}: learn

> {{DESC_LEARN}}

**Trigger:** the user wants to genuinely master a skill or topic — not just gather facts (that is [research](research.md)) — and asks to be tested, taught, drilled, or have their understanding pressure-tested. Phrases like "help me actually learn X", "build me a learning plan", "test whether I really understand X", "I think I already know X, prove me wrong", or "let me explain X back to you".

Pick the technique that fits where the user is:

- **Learning path** — they have a concrete goal and deadline and need a route to it.
- **Gap detector** — they believe they already know the topic and want that belief stress-tested.
- **Feynman check** — they just studied something and want to find out whether it actually stuck.

## Procedure

First, name the technique you are using and why, so the user knows the contract. Then run it. Throughout, do not flatter: shallow answers get told they are shallow, directly.

### Learning path
1. Pin the *real* goal, not the subject: the specific result wanted, the deadline, and what the user already masters. Ask once if any of the three is missing.
2. Build a day-by-day path to that result. Each day carries exactly three things: one task that fits the user's stated time budget (default 45 minutes), a concrete criterion so they can tell whether they did it correctly, and one explicit *what not to do* that day so time is not wasted.
3. Trace the whole path against the goal. If finishing every day would not actually deliver the result, rebuild it until it does — do not ship a path that only teaches the subject "in general".

### Gap detector
1. Take the skill the user claims to master. Set the contract: you will try to prove them wrong, and you will not go easy.
2. Ask five questions that look simple but separate someone who has truly gone deep from someone who has only skimmed — edge cases, "why", the assumption under the rule, the case where the standard answer breaks.
3. For every answer the user gives, say plainly what it reveals about what is still missing in their foundation. If an answer is shallow, name it as shallow and say what depth would have looked like.
4. Close with the honest verdict: the specific gaps to fill before the mastery claim holds.

### Feynman check
1. Have the user explain what they studied as if teaching a ten-year-old.
2. Interrupt the moment they (a) use jargon they cannot themselves unpack, (b) skip a step in the reasoning, or (c) simplify so far that the explanation becomes wrong. Name which of the three it was each time.
3. At the end, tell them exactly what those stumbles reveal about what is not yet solid in their understanding, and what to restudy.

## Done when
- The chosen technique ran to its end and the user has a concrete, honest output: a goal-tested path, a verdict on their real gaps, or a list of what is not yet solid — never empty praise.
