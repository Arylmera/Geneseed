# Docs menu — in-app documentation surface

**Date:** 2026-06-13
**Status:** approved

## Problem

Geneseed ships a substantial body of documentation — `README.md`, `SETUP.md`,
`DESIGN.md`, the OpenCode + Claude Code adapter readmes, the memory and notebook
convention readmes, and the 20+ dated specs under `docs/specs/` — but the web UI
exposes none of it. A new user lands on the dashboard with no on-ramp; an
operator has to leave the app and read the repo to learn what a command does,
what an emit mode means, or which spec covers a feature. The harness already
*reads* and renders markdown for the Library; the same engine can carry a
proper documentation surface.

## Fix

A single **Docs** entry in a new **Learn** rail group, with a left sub-nav
*inside* the page (so the rail stays calm). Content is mostly auto-discovered
from on-disk markdown — drop a new spec into `docs/specs/` and it appears —
plus a handful of *generated* pages (CLI reference parsed from `harness.py`'s
argparse, theme-aware glossary read from the deployed theme's JSON, specs
index) that read live from the harness itself, so docs can't drift from code.

### Menu structure

Ten sub-sections, grouped vertically in the left sub-nav:

1. **Get Started** — *what is Geneseed?*, quickstart, pick a path, pick a theme,
   verify your install. Pulled from `README.md` with the row that matches the
   currently deployed install highlighted (state-aware overlay).
2. **Concepts** — the mental model, one page per piece (harness model, rules,
   agents, skills, memory, notebook, wiki, context, themes, plugins). Each
   concept page is a short curated blurb followed by a "Browse the catalog →"
   link into the Library filtered to that section.
3. **CLI reference** — generated. Every subcommand of `geneseed` /
   `geneseed.cmd` / `geneseed.ps1` (`build`, `doctor`, `setup`, `tui`, `web`,
   `diff`, `upgrade`, `sync-self`, `link`, `unlink`, `context`, `learn`,
   `prompt`, `status`, `version`, `bootstrap`, `uninstall`, `git-gate`, `menu`,
   `home`) with synopsis, flags, examples. Source: parses `harness.py`'s
   `argparse.ArgumentParser` at request time so the page can't drift.
4. **Setup paths** — full `SETUP.md`, rendered. Each path heading collapsible;
   the current path opens by default (state-aware). Copy-to-clipboard on every
   command block.
5. **Adapters** — `adapters/opencode/README.md`,
   `adapters/opencode/GLOBAL-HARNESS-SPEC.md`,
   `adapters/opencode/HOW-OPENCODE-LOADS.md`, `adapters/claude-code/README.md`.
6. **Authoring & advanced** — editing laws/agents/skills under `src/`, building
   and re-theming, the self-improvement loop, the diff/back-port workflow, the
   workflow primitive, writing a new theme.
7. **Troubleshooting / FAQ** — common doctor problems and what they mean, "agent
   isn't loading docs" / "plugin not registering" / "Windows PATH didn't
   update", and other recurring questions.
8. **Design & history** — `DESIGN.md`, plus a generated index of every file
   under `docs/specs/`. Each spec is rendered on click; the index is sortable
   by date and shows the spec's first-paragraph purpose line.
9. **Glossary** — theme-aware. Definitions for the invented terms
   (*gene-seed*, *harness*, *charter*, *sigil*, *benediction*, *dictate*,
   *adept*, *rite*, *council*, *voice*, *structure*, …) with the **deployed
   theme's word** and the plain-English equivalent shown side by side, so an
   `imperial` install reads naturally and a newcomer still gets the neutral
   term. Source: each theme's JSON `LAWS`/`AGENTS`/`SKILLS`/`MEMORY`/… tokens.
10. **About** — version (from the install marker), license, repo link,
    contributing pointer.

### Backend

`rituals/web.py` gains five read-only endpoints:

- `GET /api/docs` — returns the menu structure: groups + pages with `id`,
  `title`, and `kind` (one of `markdown`, `cli`, `specs`, `glossary`,
  `concept`, `about`). Pure: a constant in code plus a discovery pass over
  `docs/specs/` so new specs appear automatically.
- `GET /api/docs/page/<id>` — returns one page. For `markdown` pages, reads the
  source file and returns its body. For `cli`, parses `harness.py`'s argparse
  via `argparse.ArgumentParser.parse_args(['--help'])` introspection and
  returns a structured `commands: [{name, help, args, opts}]` list. For
  `specs`, returns the sorted spec list. For `glossary`, loads the deployed
  theme's JSON token map plus the neutral map for the side-by-side view. For
  `concept`, returns inline markdown from the registry.
- All endpoints reuse `WebState` (cached inventory) and the same `_within`
  guard the library uses — a `markdown` page can only resolve a source file
  inside the source tree.

### Frontend

- New rail group **Learn** with a single **Docs** entry; route `#/docs` and
  `#/docs/<page-id>`.
- New page `web/src/pages/Docs/`:
  - `index.jsx` — shell with left sub-nav (groups + pages) and right content
    pane. Selection drives the route; the topbar's `query` filters the visible
    sub-nav entries (search scoped to docs).
  - `MarkdownPage.jsx` — renders a `markdown`/`concept` page using the existing
    `Markdown` component with two additions: (a) an in-page TOC generated from
    the rendered `<h2>`/`<h3>` headings, with anchor scrolling; (b) for
    `markdown` pages whose `id` matches a known state-aware overlay (e.g.
    setup paths, install modes), the row that matches the deployed install is
    visually highlighted.
  - `CliPage.jsx` — renders the generated CLI structure: each command as a
    card with `synopsis`, args, options, and copy-to-clipboard examples.
  - `SpecsIndex.jsx` — renders the sorted spec list, click-through to a
    `MarkdownPage` view of the spec.
  - `Glossary.jsx` — renders the theme-aware table.
- New API module `web/src/api/docs.js`, exporting `docs()`, `docsPage(id)`.

### Cross-cutting

- **Search scoped to docs** — the topbar `query` already exists; on the Docs
  page it filters the sub-nav.
- **"Try this" buttons** — wherever the docs reference a safe action
  (`run doctor`, `open diff`, `open themes`, `rebuild`), surface a button that
  calls the existing `runAction()` hook.
- **Cross-link out** — markdown links to `[[<name>]]` resolve as wikilinks via
  the same `_resolve_links` path the Library uses, so a docs page can deep-link
  into an agent or skill.

## Non-goals

- **Editing from the UI.** Docs are read-only; back-porting changes goes
  through the existing diff / improvements-export flow.
- **External links / web fetch.** Everything is local — no network reads at
  request time.
- **Per-page authoring metadata.** The registry is a plain list; ordering is
  by registry order.

## Why this shape

A single rail entry keeps the chrome calm. The heavy lifting lives inside the
page where it has room. The split between *auto-discovered* (markdown files,
specs) and *generated* (CLI, glossary, about) means most of the surface
maintains itself: edit a markdown file or a theme JSON or add a spec, and the
docs follow without UI changes. The few hand-curated pages (Concepts,
Authoring, Troubleshooting) are short blurbs anchored to durable code
artifacts, not duplicated reference material, so they're cheap to keep current.
