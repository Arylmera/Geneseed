# Activity v1.1 — enriched session cards

**Date:** 2026-06-18
**Status:** ready to build
**Parent:** [[2026-06-15-live-activity-surface]] (v1 — flat status cards, shipped)

## Problem

v1 ships a flat card per live OpenCode session: cwd · title · agent · status ·
relative time. It answers "is a session alive and is it busy" but not the
question every comparable tool (LangSmith, AgentOps, Devin, Cursor, the codex
pet-overlay, Claude Code's TUI) leads with: **what is it doing right now, how
much has it cost, and what has it changed.**

The decisive finding from the design review: the writer already receives the
events that carry this — it reads `mode`/`cwd`/`title` off `message.updated` and
discards everything else, and never switches on `message.part.updated`'s
`part.type`. So this is a *read-more* change, not new plumbing. No new hooks, no
polling, no control channel — the passive, crash-isolated, one-file-per-session
seam is unchanged.

## Scope (locked with the maintainer)

Render, per session card, all of:

1. **Live phase headline** — the current action ("Editing Activity.jsx",
   "Running tests", "Thinking"), replacing bare busy/idle as the lead line.
2. **Token · cost · elapsed · model** counters.
3. **Files touched + churn** — count and +adds/−dels, with a short file list.
4. **Todo / plan checklist** — `done/total` with a progress strip.
5. **Blocked / error as first-class status** — a distinct status + colour for
   awaiting-permission and for an errored turn, not folded into idle.

Read-only. **Out of scope:** control actions (interrupt/approve), context-window
% (its limit is not on any event — needs `client.config`), the session detail
page (→ [[2026-06-18-activity-v1.2-session-detail]]), and the sub-agent
parent→children tree (still deferred).

## Data — every field is on an event we already receive or one `case` away

All line numbers are `~/.config/opencode/node_modules/@opencode-ai/sdk/dist/gen/types.gen.d.ts`.

| Card element | Source event → field | Lines | Cost |
|---|---|---|---|
| phase headline | `message.part.updated` → `ToolPart.state` (`ToolStateRunning.title` ?? `tool`); `ReasoningPart` → "Thinking" | 218–230, 263–270, 158–171 | switch on `part.type` |
| model | `message.updated` → `AssistantMessage.modelID` | 108 | already read the message |
| tokens (session) | `AssistantMessage.tokens.{input,output}` summed per assistant-message id | 117–125 | accumulate (below) |
| cost (session) | `AssistantMessage.cost` summed per assistant-message id | 116 | accumulate (below) |
| elapsed (turn) | `AssistantMessage.time.created` while busy → UI ticks `now − start` | 102–105 | field on the message |
| files + churn | `session.updated` → `Session.summary.{files,additions,deletions,diffs}`; `FileDiff.{file,additions,deletions}` | 470–475, 32–38 | new `case` |
| todos | `todo.updated` → `EventTodoUpdated.properties.todos` (`Todo.{content,status}`) | 431–455 | new `case` |
| blocked (status) | `permission.updated` (pending) ↔ `permission.replied` (clear) | 384–395 | new `case` |
| error | `AssistantMessage.error` ?? `ToolStateError.error` ?? `session.error` | 106, 248–253, 518–524 | read on events we get + one `case` |

**Token/cost accumulation.** `cost`/`tokens` are per-`AssistantMessage`, and
`message.updated` fires repeatedly for the *same* streaming message (same `id`,
growing numbers). So the writer keeps an off-entry map per session,
`messageId → {cost, tokens}`, updated in place on each `message.updated`, and
writes the session totals as the sum. This is correct across both streaming
(same id overwrites) and multiple turns (new id adds). The map is in-memory
only — never serialized.

## Writer — `geneseed-activity.js`

Extend the existing `event` switch. New/expanded cases:

- `message.updated` (assistant): capture `modelID` → `model`; update the
  accounting map from `cost`/`tokens` → set entry `cost`/`tokens`; capture
  `time.created` → `turn_started_at` (cleared on idle/waiting); set `error` from
  `error` if present.
- `message.part.updated`: switch on `part.type` —
  - `tool` + `state.status==="running"` → `phase = state.title || tool`
  - `tool` + `state.status==="error"` → `error = state.error`
  - `reasoning` → `phase = "Thinking"`
  - (text streaming already keeps status busy)
- `session.updated`: read `info.summary` → `files = {count, additions, deletions,
  items: diffs.slice(0, FILE_CAP).map({file,additions,deletions})}`.
- `todo.updated`: `todos = {done: count(status==="completed"), total,
  items: todos.slice(0, TODO_CAP).map({content,status})}`.
- `permission.updated` → `status="blocked"`, `blocked_on = title`;
  `permission.replied` → recompute status (back to busy).
- `session.error` → `error`.

`session.idle` clears `turn_started_at` and `phase` (turn over). Status
precedence: `blocked` > `busy`/`waiting-input`/`idle`. `error` is an orthogonal
field (a red sub-badge) — it does not replace status, so you still see whether
the errored session is now waiting on you.

**Per-session entry schema (v1.1 — additive, all optional so the reader tolerates
a v1 writer):**
```jsonc
{
  "session_id": "ses_…", "agent": "build", "title": "…", "cwd": "/abs",
  "status": "busy" | "waiting-input" | "idle" | "blocked",
  "pid": 12345, "updated_at": 1718…,
  "model": "claude-opus-4-8" | null,
  "phase": "Editing Activity.jsx" | "Thinking" | null,
  "turn_started_at": 1718… | null,            // epoch s; UI ticks live elapsed
  "cost": 0.62,                               // session total $
  "tokens": 48213,                            // session total in+out
  "files": { "count": 3, "additions": 124, "deletions": 18,
             "items": [ { "file": "web/src/pages/Activity.jsx", "additions": 60, "deletions": 4 } ] },
  "todos": { "done": 3, "total": 5,
             "items": [ { "content": "wire the reader", "status": "completed" } ] },
  "blocked_on": "bash: rm -rf build/" | null,
  "error": "tool failed: …" | null
}
```
Caps: `FILE_CAP=8`, `TODO_CAP=12` (bound the file size; the detail page in v1.2
carries the full lists). Toggle/disable behaviour is unchanged from v1.

## Reader — `_web_activity.py`

`_activity_entries` passes the new fields through verbatim (it already builds a
normalized dict per file). Defaults for a v1 writer: `phase/model/error/blocked_on
→ None`, `cost/tokens → 0`, `files/todos → None`. No new endpoint. Pruning,
gating, and the toggle are unchanged.

## UI — `web/src/pages/Activity.jsx`

Rework `SessionCard` to the enriched layout (mockup approved 2026-06-18). Pure
reuse of existing tokens — `.card`, `.badge` variants, `.dim`/`.mono`, the
`.feed-dot` pulse — plus one small set of card-internal rows:

- **Header:** pulse dot · title · cwd (muted) · model pill · status badge.
- **Phase line:** an icon + the `phase` string (the lead signal). Omitted when
  null (idle).
- **Counter row:** `tokens` (compact, e.g. 48.2k) · `$cost` · live `elapsed`
  (ticks from `turn_started_at` via a 1s local interval while busy) · files
  `N +adds −dels`.
- **Footer row:** todo strip `●●●○○ done/total` (left) · `error` red sub-badge or
  `blocked_on` label (right) when present.

Status → colour: `busy`→accent ("working"), `waiting-input`→warn ("your move"),
`blocked`→a distinct transitional colour ("blocked"), `idle`→dim. `error` is a
red `.badge.bad` sub-badge regardless of status. (k9s' rule: transitional ≠
error; both ≠ success.)

The left accent border encodes status colour (mockup). Cards remain a flat
`.stack.gap-12` list; clicking does nothing yet (→ v1.2 detail).

## Status / colour vocabulary (locked, reused by v1.2)

| status | label | class | meaning |
|---|---|---|---|
| busy | working | `.badge.acc` (accent) | agent is acting |
| waiting-input | your move | `.badge.warn` | turn done, needs you |
| blocked | blocked | `.badge` transitional | awaiting a permission |
| idle | idle | `.badge` dim | alive, quiet |
| (error) | — | `.badge.bad` sub-badge | last turn/tool errored |

## Why this shape

- **No new architecture.** Every field rides events the writer already receives
  or one `case` on `part.type`. The seam stays passive/crash-isolated/no-RPC.
- **Glance value of the whole field, cheaply.** Phase + cost + files + plan is
  ~90% of what LangSmith/Devin/Cursor/AgentOps surface, without their backends.
- **Forward-compatible.** The schema is additive and optional; v1.2's detail page
  reuses the same status vocabulary and the same files/todos data (uncapped).
