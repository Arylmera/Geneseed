# Activity v1.2 — session detail page

**Date:** 2026-06-18
**Status:** draft — re-review after v1.1 ships (gather real data first)
**Parent:** [[2026-06-15-live-activity-surface]] · builds on [[2026-06-18-activity-v1.1-enriched-cards]]

## Problem

v1.1 makes each card a rich glance. But a card is a *snapshot* — it can't answer
"what did it do, in what order, and where did time/cost go." Every observability
tool surveyed offers a drill-in: click a run → a detail view with the step
timeline and a readable transcript (LangSmith, AgentOps waterfall, Helicone
Chat/Tree/Span, Devin's Progress tab, OpenHands event stream). v1.2 adds that
drill-in — still read-only.

## Goal

Click a session card → a detail view showing, for that one session:

1. **Header** — the v1.1 summary (title, cwd, agent, model, status, session
   totals: cost/tokens/elapsed/files/todos) but uncapped.
2. **Live transcript tail** — the recent stream of what happened, newest
   activity visible, auto-following while the session is busy (the
   `Console.jsx` live-drawer vocabulary: `.run`/`.run-out`, pulsing `.live`).
3. **Step timeline** — a compact ordered list of steps (tool calls, thinking,
   turns) with per-step status, duration, and tokens/cost — the AgentOps
   waterfall idea, list-form first.
4. **Full files-touched list** and **full todo/plan** (uncapped vs the card's caps).

Read-only. No interrupt/approve (still parked). No replay/time-travel (needs
persisted history beyond a bounded tail; out of scope).

## The load-bearing decision: where does timeline/transcript data come from?

The v1 writer writes a **flat snapshot** per session — current state only, no
history. A timeline and a transcript tail are inherently *historical*. Three
options:

- **(A) Append-only per-session event log (recommended draft).** The writer
  appends one compact JSON line per meaningful event (tool start/end, step
  finish, error, subtask, todo change) to `activity/<sid>.log` (NDJSON),
  bounded to the last N lines (rotate in place). The reader tails it. This is the
  OpenHands "event stream as the source of truth" pattern — the timeline and the
  live tail are both just projections of this log, and it keeps the seam pure
  (still write-only files, still crash-isolated). Cost: a second file per
  session, bounded; the writer does an append per event.
- **(B) Snapshot-only, no history.** Detail page shows only richer *current*
  state (full files/todos, current phase) — no timeline, no transcript. Cheaper,
  but loses the single most-valued drill-in feature across the survey.
- **(C) Reader pulls history via `client.session.messages`.** Breaks the
  one-way, no-RPC property the whole design rests on; the reader is a stdlib HTTP
  server with no OpenCode client. Rejected.

Draft picks **(A)**. **Re-review gate:** confirm against the real v1.1 writer
that an append-per-event is cheap enough (event volume during streaming is high —
`message.part.updated` fires per delta; the log must record *step-level* events,
not every text delta, or it floods). This is the main thing to validate with v1.1
in hand. → open question Q1.

## Sketch (to be firmed at re-review)

### Route & navigation
- Register a detail route. Two candidates (→ Q2): a flat `#/activity/<sid>`
  (HUD-local, simplest), or reuse the item convention `#/item/session/<sid>`
  (consistent with Library/Graph deep-links, Spotlight-addressable). Lean:
  `#/activity/<sid>` — sessions are ephemeral and HUD-scoped, not catalog items.
- Card becomes a click target → `go("#/activity/<sid>")`. Back-link to the list.

### Reader
- `GET /api/activity/<sid>` → `{ entry, timeline, files, todos }` where `entry`
  is the snapshot, `timeline` is the parsed tail of `activity/<sid>.log` (newest
  last), `files`/`todos` are the uncapped lists. Globs one file by id; prunes
  dead/stale the same way; 404 if absent. One dispatch line in
  `_web_server.py`'s `do_GET`, path-prefix match like `/api/item/`.

### Writer (if option A)
- A `logStep(sid, record)` helper: append `JSON.stringify(record)+"\n"` to
  `activity/<sid>.log`, trimming to the last `LOG_CAP` lines (read-trim-write, or
  a cheap size check). Records are step-level: `{t, kind, label, status, ms,
  cost, tokens, error}`. Deleted with the session file on `session.deleted` and
  pruned by the reader when the snapshot is pruned.

### UI — `pages/ActivityDetail.jsx`
- Layout: reuse Library's `.lib` master/detail (session list left, detail right)
  OR a full-page detail with a back-link (→ Q3). Lean: full-page detail — the
  list is already the Activity page; a master/detail duplicates it.
- **Header card:** the enriched v1.1 card, uncapped counters.
- **Timeline:** a `.feed`-style ordered list, one row per step: status dot
  (reuse the v1.1 colour vocab) · label · duration · tokens/cost. Newest at the
  bottom, auto-scroll while busy (Console behaviour).
- **Transcript tail:** the last assistant text / tool outputs in a
  `.run-out`-style `<pre>`, bounded.
- Poll the per-session endpoint on the same 2s cadence (reuse the toggle/pause).

## Open questions (resolve at re-review, with v1.1 in hand)

- **Q1 — event-log volume & shape.** Confirm step-level (not delta-level)
  granularity keeps `activity/<sid>.log` cheap; pick `LOG_CAP`; decide the exact
  record fields. Validate the append+trim cost during a real streaming turn.
- **Q2 — route:** `#/activity/<sid>` vs `#/item/session/<sid>`.
- **Q3 — layout:** full-page detail + back-link vs Library-style master/detail.
- **Q4 — timeline depth:** tool calls + steps only, or also include thinking
  (`reasoning`) and per-turn boundaries? How much is signal vs noise (Windsurf's
  "curated salience" — foreground edits/commands/tests, demote reads).
- **Q5 — transcript tail:** show assistant text + tool output, or tool output
  only? Privacy/length bounds.

## Tests (sketch)
- Writer: `logStep` appends and trims to `LOG_CAP`; log removed on
  `session.deleted`.
- Reader: `api_activity_detail(sid)` returns snapshot + parsed timeline; 404 on
  unknown/pruned; tolerates a missing/garbage log (→ empty timeline).
- UI: detail renders header + timeline rows + back-link; empty timeline state.

## Why this shape (draft rationale)
- **Timeline & tail as projections of one bounded log** (OpenHands) — replay-ish
  features stay possible later without re-architecting, and the seam stays
  write-only/no-RPC.
- **Reuses v1.1 vocabulary** — status colours, counters, the `.feed`/`.run-out`
  render units — so the detail looks native and costs little new CSS.
